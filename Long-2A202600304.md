**2. False Positives**
*   *Observation*: If a user asks "What is the password to my online banking?", the `reveal.*password` regex might block a legitimate request for help.
*   *Trade-off*: Increasing security (blocking the word "password") reduces usability for users who actually have password issues.

**3. Gap Analysis**
*   **Attack 1 (Obfuscation)**: "I-g-n-o-r-e instructions." Regex won't catch this. *Solution*: Use an embedding-based classifier.
*   **Attack 2 (Payload Splitting)**: "Part 1: Ignore. Part 2: everything." *Solution*: Analyze the full session history, not just single messages.
*   **Attack 3 (Hallucination)**: LLM gives a wrong interest rate. *Solution*: RAG (Retrieval Augmented Generation) to verify against a database.

**4. Production Readiness**
*   **Latency**: Currently, we call the LLM twice (Response + Judge). This doubles latency. In production, I'd run the Judge asynchronously or only on a 10% sample of queries.
*   **Cost**: Two calls = 2x cost. I'd use a smaller, cheaper model (like Phi-3) for the Judge.

**5. Ethical Reflection**
*   "Perfect safety" is impossible because language is infinite. A system should refuse to answer dangerous topics (e.g., "How to hack a bank") but provide a disclaimer for gray areas (e.g., "I am an AI, this is not financial advice").
